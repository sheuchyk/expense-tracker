# Правила код-ревью

Документ описывает правила и чеклист для ревью кода в проекте expense-tracker-my (Next.js 16 + NestJS 11 + Prisma 6, монорепо на npm workspaces).

## 1. Общие принципы

- Ревью — обязательный шаг перед merge в `master`. Merge только после апрува и зелёного CI.
- Ревьюер оценивает корректность, читаемость, безопасность и соответствие архитектуре, а не личные предпочтения по стилю.
- Автор PR обязан самостоятельно пройтись по чеклисту до запроса ревью.
- Комментарии оставляем предметно: цитата кода → проблема → предлагаемое решение.
- Уровни замечаний: **must-fix** (блокирует merge), **should-fix** (желательно поправить), **nit** (стилистика, необязательно).

## 2. Формат Pull Request

- Целевая ветка — всегда `master`. Ветка создаётся от свежего `master` с именем `<type>/<short-kebab-case>` (`feat/`, `fix/`, `refactor/`, `docs/`, `chore/`).
- Заголовок PR соответствует Conventional Commits: `feat(scope): short imperative subject` (≤ 72 символа).
- Тело PR содержит секции **Summary**, **Changes** (Backend / Frontend), **Test plan** (см. CLAUDE.md).
- PR короткий и сфокусирован на одном изменении. Большие PR разбиваем на серию.
- Merge — **squash merge**; сообщение squash-коммита тоже в Conventional Commits.
- После merge ветка удаляется локально и на remote.

## 3. Коммиты

- Формат: `<type>(<scope>): <subject>`.
- `subject` — в повелительном наклонении, в нижнем регистре, без точки в конце, ≤ 72 символа.
- Допустимые типы: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `chore`.
- Дополнительный контекст — в теле коммита через пустую строку.

## 4. Архитектура фронтенда (Feature Slice Design)

- Импорты строго вниз по слоям: `app → views → widgets → features → entities → shared`. Обратные и «горизонтальные» импорты внутри одного слоя запрещены.
- `src/app/` содержит только тонкие Next.js-роуты — импорт и рендер компонента из `views/`.
- `features/*` не импортируют друг друга. Общий код выносится в `entities/` или `shared/`.
- Структура слайса: `ui/`, `api/`, `model/`. Не смешивать бизнес-логику и представление.
- shadcn/ui-компоненты живут в `shared/ui/<component-name>/index.tsx`.
- Дизайн-токены — CSS-переменные в `globals.css`, маппинг в `tailwind.config.ts`. Не хардкодим цвета/радиусы в компонентах.

## 5. Архитектура бэкенда (NestJS)

- Все эндпоинты — под префиксом `/api` (задаётся в `src/main.ts`).
- Модули структурированы по фичам; используем DI, а не прямые импорты сервисов между модулями без объявления в `providers`.
- Валидация входа — через DTO и class-validator. Никаких «сырых» body/query в контроллерах.
- Работа с БД — только через Prisma. Не пишем SQL-строки без веских причин.
- Ошибки — через встроенные исключения Nest (`NotFoundException`, `BadRequestException`, …), не бросаем `new Error()` из контроллеров/сервисов.

## 6. База данных и Prisma

- Любое изменение схемы сопровождается миграцией (`npm run prisma:migrate`) и обновлённым клиентом (`npm run prisma:generate`).
- Миграции не редактируются постфактум — только новые.
- Индексы и уникальные ограничения проверяем для полей, по которым идут запросы/поиск.
- Не логируем и не возвращаем в API секретные поля (пароли, токены).

## 7. TypeScript и качество кода

- Никаких `any` без явного обоснования в комментарии. Предпочитаем `unknown` + narrowing.
- Публичные функции/хуки/DTO имеют явные типы возвращаемых значений и параметров.
- Не отключаем правила ESLint/TS локально без комментария с причиной.
- Именование: `camelCase` для переменных/функций, `PascalCase` для компонентов/типов, `SCREAMING_SNAKE_CASE` для констант окружения.
- Комментарии — только там, где неочевидно **почему**. Что делает код, должно быть понятно из имён.

## 8. Безопасность

- Секреты и `DATABASE_URL` — только в `.env.local`, никогда не в коде и не в коммитах.
- Пользовательский ввод валидируется на бэкенде (DTO + class-validator), даже если фронт уже валидирует.
- Проверяем OWASP Top 10: SQLi (Prisma параметризует, но следим за `$queryRaw`), XSS (не используем `dangerouslySetInnerHTML` без санитайзинга), CSRF, авторизация на каждом защищённом эндпоинте.
- Ошибки не «протекают» наружу со стеком/деталями БД — только безопасные сообщения.

## 9. Тестирование и ручная проверка

- В PR-описании обязателен раздел **Test plan** с шагами проверки happy path и минимум одного edge case.
- Автор проверяет изменение локально: `npm run dev:backend` + `npm run dev:frontend`, проходит фичу в браузере.
- Для изменений в API — проверяем запрос/ответ (curl / DevTools / Postman).
- Ломающие изменения снабжаются пометкой `BREAKING CHANGE:` в теле коммита/PR.

## 10. Линт, формат и сборка

- Перед пушем: `npm run lint` и `npm run format` — оба зелёные.
- Сборка обеих частей должна проходить: `npm run build:frontend` и `npm run build:backend`.
- Не добавляем зависимости «на всякий случай». Каждая новая зависимость обоснована в описании PR.

## 11. Чеклист ревьюера

- [ ] Заголовок и тело PR соответствуют формату (Conventional Commits, секции Summary/Changes/Test plan).
- [ ] Ветка соответствует именованию `<type>/<short-kebab-case>` и создана от свежего `master`.
- [ ] Изменение сфокусировано: один PR — одно логическое изменение.
- [ ] Соблюдены слои FSD на фронтенде; нет запрещённых импортов.
- [ ] На бэкенде соблюдены модульность Nest, DI, DTO-валидация, использование Prisma.
- [ ] Нет утечек секретов и логирования чувствительных данных.
- [ ] TypeScript строгий: нет немотивированных `any`, публичные типы явные.
- [ ] Миграции Prisma присутствуют и согласованы со схемой.
- [ ] Обновлены/добавлены тесты или указан осмысленный ручной test plan.
- [ ] Линт, формат и сборка проходят локально и в CI.
- [ ] Комментарии в коде объясняют **почему**, а не **что**.
- [ ] Удалён мёртвый код, отладочные `console.log`, TODO без владельца.

## 12. Как автору отвечать на ревью

- Отвечаем на каждый комментарий: либо фиксом, либо аргументом, почему оставляем как есть.
- Правки — отдельными коммитами в ту же ветку; финальный merge всё равно squash.
- Резолвим тред только после того, как ревьюер подтвердил решение (или это `nit`).
