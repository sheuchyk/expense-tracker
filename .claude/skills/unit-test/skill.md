# Unit-Test Skill

Сгенерируй unit-тесты для файла, переданного пользователем.

## Аргументы

Скилл принимает один аргумент:

1. **Путь к файлу** — относительный или абсолютный путь к исходному файлу, для которого нужно написать unit-тесты (например, `backend/src/users/users.service.ts`).

Если аргумент не передан — попроси пользователя указать путь к файлу.

## Требования к тестам

- На **TypeScript**, с использованием **Jest** и `@nestjs/testing` (если тестируется NestJS-класс).
- Все внешние зависимости замоканы через `jest.fn()` / `Test.createTestingModule` с `useValue`.
- Каждый публичный метод покрыт как минимум тремя типами кейсов:
  - **Happy path** — корректный сценарий выполнения.
  - **Edge cases** — граничные значения, пустые массивы/строки, `null`/`undefined`.
  - **Error cases** — ожидаемые исключения (`NotFoundException`, `UnauthorizedException`, ошибки Prisma и т. п.).
- Имена тестов на **русском языке** в формате `it('должен ...', ...)`.
- Файл теста называется `[name].spec.ts` и лежит **рядом** с тестируемым файлом.

## Алгоритм выполнения

1. Прочитай целевой файл по переданному пути (`Read`).
   Если файла нет — сообщи пользователю и остановись.

2. Проверь, что в проекте установлены нужные зависимости:
   ```bash
   cat backend/package.json
   ```
   Убедись, что присутствуют `jest`, `@types/jest`, `ts-jest`, `@nestjs/testing`.
   Если чего-то не хватает — установи:
   ```bash
   npm install --workspace backend --save-dev jest @types/jest ts-jest @nestjs/testing
   ```

3. Определи публичный API файла:
   - Все `public` методы класса (методы без модификатора считаются публичными).
   - Конструкторные зависимости (для моков).
   - Импорты внешних сервисов, репозиториев, `PrismaService`, `JwtService` и т. п.

4. Для каждого публичного метода сформируй набор тест-кейсов:
   - **Happy path** — успешное выполнение с валидными входными данными.
   - **Edge cases** — пустые/граничные значения, отсутствие опциональных параметров.
   - **Error cases** — проверка проброса/трансформации исключений.

5. Замокай все зависимости через `jest.fn()`. Для NestJS используй `Test.createTestingModule({...}).compile()` с `providers` и `useValue`.

6. Создай файл `[name].spec.ts` рядом с тестируемым файлом (`Write`).
   Шаблон структуры:
   ```ts
   import { Test, TestingModule } from '@nestjs/testing';
   import { <ClassName> } from './<file-name>';

   describe('<ClassName>', () => {
     let service: <ClassName>;
     // моки зависимостей

     beforeEach(async () => {
       const module: TestingModule = await Test.createTestingModule({
         providers: [
           <ClassName>,
           { provide: <Dep>, useValue: { <method>: jest.fn() } },
         ],
       }).compile();

       service = module.get<<ClassName>>(<ClassName>);
     });

     describe('<methodName>', () => {
       it('должен ... (happy path)', async () => { /* ... */ });
       it('должен ... (edge case)', async () => { /* ... */ });
       it('должен бросить ... (error case)', async () => { /* ... */ });
     });
   });
   ```

7. Запусти тесты:
   ```bash
   npm run test --workspace backend -- <file-name>.spec
   ```

8. Если тесты упали:
   - Проанализируй вывод Jest.
   - Исправь тест (или мок), не меняя исходный файл.
   - Перезапусти тесты. Повтори до зелёного прогона (максимум 3 итерации, затем сообщи пользователю).

9. Верни пользователю:
   - Путь к созданному файлу тестов.
   - Список покрытых методов и типов кейсов.
   - Итог прогона (passed/failed).

## Важные ограничения

- **Не меняй** тестируемый файл — только создавай/правь `*.spec.ts`.
- **Не мокай** то, что тестируется напрямую (сам класс/функция должны исполняться, моками закрываются только внешние зависимости).
- **Не используй** `any` без крайней необходимости — типизируй моки через `jest.Mocked<T>` или `Partial<T>`.
- **Не удаляй** существующий `*.spec.ts`, если он уже есть — вместо этого дополни его недостающими кейсами и предупреди пользователя.
- **Не запускай** весь тест-сьют проекта — только тесты для целевого файла.
- Если файл не содержит тестируемой логики (только типы/интерфейсы/DTO без валидации) — сообщи пользователю, что тесты не нужны, и не создавай пустой spec.
